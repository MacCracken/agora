# 2026-05-23 — agora 0.7.0 pre-1.0 security audit

> **Audit window**: agora 0.6.0 (`8be27d0`) — post-M6 close.
> **Scope**: full IAC parser, post storage + ingress filter, M6 auth surface (sigil challenge/response, From-header parsing, per-board policy), session lifecycle, CLI input validation.
> **Method**: line-by-line read of `src/telnet.cyr` (718), `src/board.cyr` (956), `src/account.cyr` (497), `src/main.cyr` (1612) against CLAUDE.md "Security Hardening" checklist; review of CVE-2020-10188, CVE-2011-4862; cross-check against ADR 0006 threat model.
> **Baseline**: clean build (`cyrius build src/main.cyr build/agora`), 70/70 tests pass.

---

## Severity rubric

- **CRITICAL**: remote compromise / privilege escalation from a single bad input.
- **HIGH**: remote DoS or auth bypass with moderate effort, or data corruption from untrusted input.
- **MEDIUM**: requires specific conditions (operator misconfiguration, concurrent-accept refactor, very long uptime).
- **LOW**: defense-in-depth — current code is correct, but a near-future change could regress.
- **DOCUMENTED**: known trade-off captured in an ADR; surfaced here for the record, not actionable.

---

## Findings

### CRITICAL

*None.* No remote-compromise paths found in 0.6.0. The IAC parser bounds-checks every state transition, the auth flow uses fresh single-use nonces via `/dev/urandom`, and post storage uses `O_EXCL` + per-board flock for the claim-and-write critical section.

### HIGH

#### H1 — CRLF / header injection via CLI `--subject`

**Where**: `src/main.cyr:1228` (`parse_subject_flag`), `src/board.cyr:493` (`post_format_with_headers`).

A subject string from `--subject "Foo\r\nReply-To: 42\r\nFrom: alice 0000000000000000"` (literal CRLF bytes after shell quoting) is written directly into the post file's header block with no per-byte validation. Downstream `post_reply_to` / `post_from` then parses the injected lines as real headers, producing a post that appears to be from `alice` with a forged Reply-To threading.

The wire-side `post` flow is safe because the line buffer terminates on CR/LF; CLI is the only injection vector.

**Why this is HIGH, not LOW**: although the CLI is operator-trusted at v1.0 LAN scale, the **stored post** persists with forged headers. Any future tool that reads the store (federation pillar in v2.x; backup / migration scripts; doc-extraction) trusts the headers. A single bad CLI invocation corrupts the audit trail permanently.

**Fix**: validate subject bytes at `cmd_post` entry — reject any byte < 0x20 except TAB. Mirror the same filter that `input_byte_ok` applies on the wire (drop NUL + ESC), extended to drop CR + LF for header-context inputs.

**Repro**: `echo "body" | ./build/agora post --subject $'foo\r\nReply-To: 1'`; `./build/agora read <id>` shows two headers; `post_reply_to` returns `1`.

#### H2 — `--board` path-traversal in CLI `list` / `read` / `post` (operator surface)

**Where**: `src/main.cyr:1321` (`cmd_list`), `:1364` (`cmd_read`); `cmd_post` validates correctly at `:1273`.

`cmd_list` and `cmd_read` accept `--board <name>` without calling `board_name_valid`. A `--board "../../etc"` resolves via `board_path(store, "../../etc", ...)` → `<store>/../../etc` → `dir_list` of `/etc`. Read leaks. Write is not exposed via these paths.

**Why this is HIGH, not MEDIUM**: same v2.x federation reasoning as H1 — once any tool scripts agora to surface boards from a peer (untrusted) source, this becomes remote disclosure. Fix it before that consumer arrives.

**Fix**: call `board_name_valid(board)` in `cmd_list` and `cmd_read` unless `streq(board, "main") != 0`, with the same error message as `cmd_post`.

**Repro**: `./build/agora list --board "../../tmp"` — `dir_list` will enumerate `<store>/../../tmp` (silently empty if no `.txt` files, but the syscall happened).

#### H3 — `post_from` does not re-validate handle alphabet on read

**Where**: `src/account.cyr:280` (`post_from`), `src/main.cyr:431, 1389` (telnet `read` / CLI `read` render path).

`post_from` reads the `From: <handle> <fp16>` header and writes the raw handle bytes into the caller's buffer. The renderer then `send_buf(cfd, hbuf, strlen(hbuf))` directly. There is no re-validation that the handle conforms to `handle_valid`'s alphabet at read time.

Today this is safe because `account_register` only ever writes handles that have passed `handle_valid`. But:
- An operator who edits `<store>/.users/<fp>/handle` directly can inject CR / LF / ESC / arbitrary bytes.
- A future M-bite that adds federated import or backup-restore could land non-validated handles in the store.
- The render path then **emits attacker-controlled bytes to every reader's terminal** — exactly the ESC-sequence-in-post attack `input_byte_ok` already mitigates for bodies.

**Why this is HIGH**: a single byte-flip in a handle file from a compromised admin host poisons every subsequent `read` call cluster-wide. The fix is one alphabet check per render.

**Fix**: in `post_from`, after the memcpy into `handle_out`, re-run `handle_valid(handle_out)`; on fail, treat as anonymous (clear handle_out + fp_out, return 0). Same for `fp_out` (must be exactly 16 lowercase hex).

### MEDIUM

#### M1 — Unbounded bump-allocator growth in long-running `serve`

**Where**: `src/main.cyr:786-789` (per-connection identity slot allocs), `:766-789` (per-connection rxbuf 4KB + line 4KB + post 64KB + subject 256B + current_board 33B + session slots ~150B = **~73 KB / conn**), `:417, 459, 514, 1295, 1330, 1377` (per-`read` / per-`list` / per-`reply` allocs in `session_execute` and CLI verbs — **~100 KB / `read` call** worst case).

Cyrius `alloc()` is bump-allocated and never freed (per CLAUDE.md). Every connection accreted ~73 KB; every `read` inside that connection adds ~100 KB. A long-running deployment with 8 connections / hour reading 4 posts each leaks ~4.3 MB / hour. RSS climbs unboundedly until OOM.

**Why this is MEDIUM, not HIGH**: requires uptime (~weeks) and active use to materialize. Local LAN BBS at v1.0 scale won't hit it. Will become HIGH once concurrent-accept lands (the bump-arena vs per-conn-arena refactor is co-scheduled).

**Fix**: queue for the concurrent-accept refactor cycle. Per-connection memory should be allocated via a per-conn arena that's freed in bulk at `sock_close`. Per-command transient buffers (post_read scratch, list display buffers, replies_to ID array) should be in a per-call arena. **Not a 0.7.x patch — that's a v0.8 hardening item.**

#### M2 — Login challenge not bound to socket fd

**Where**: `src/main.cyr:92-93` (g_login_fp, g_login_nonce as globals), `:548-591` (`login` command), `:872-927` (MODE_LOGIN_AWAIT_SIG handler).

Today the accept loop single-tracks (`cmd_serve_on` at `:1018`), so global `g_login_*` slots are de facto per-session. Once concurrent-accept lands, two clients connecting simultaneously could collide: client A sends `login alice`, the server populates `g_login_fp = alice_fp` and sends nonce_A; client B's `login bob` overwrites `g_login_fp = bob_fp` and sends nonce_B; client A replies with sig over nonce_A; server now verifies against bob's pubkey. Misattribution → silent auth bypass for A as B (or, more likely, refuse + retry until A gives up).

**Why this is MEDIUM, not HIGH**: not exploitable today. Becomes load-bearing the moment the accept-loop refactor lands.

**Fix**: same refactor as M1 — move identity + login-challenge slots into a per-connection struct allocated in `handle_client`. Same epoch.

#### M3 — `parse_post_id` integer overflow on long digit prefixes

**Where**: `src/board.cyr:240-250`.

`n = n * 10 + (c - 48)` can overflow i64 on a filename like `<20+ digits>.txt`. Cyrius i64 wraps; overflow produces a small / negative integer. `post_max_id` then sees a small `id` and `if (id > max_id)` ignores it — `post_new` claims `max_id + 1` and the `O_EXCL` filename does not collide with the overflow-derived name, so no corruption.

**Why this is MEDIUM, not LOW**: today's behavior is correct because no caller trusts the returned ID semantically beyond ordering. But future use (e.g., a "compact post IDs" admin verb, or `board_can_post` taking a post-ID arg) could regress.

**Fix**: cap the prefix at 18 digits in `parse_post_id`; reject anything longer.

#### M4 — Anonymous session can `enter` (and auto-create) named boards

**Where**: `src/main.cyr:318-353` (`enter`), `src/board.cyr:189` (`board_ensure`).

`enter <name>` runs `board_ensure` which calls `mkdir` — anonymous (unauthenticated) clients can create directories at will under the store. Combined with the bump-alloc M1 finding, an attacker can:

1. Connect, send `enter board00001`, disconnect.
2. Repeat with `board00002`, `board00003`, ...
3. Each connection: ~73 KB process memory + one directory on the FS.

Storage exhaustion in ~10k connections (with a 32-byte name → ~100 MB inodes + dentries). Process memory exhaustion in ~10k connections (~730 MB RSS).

**Why this is MEDIUM**: requires sustained connection volume; a real deployment will notice the connections. But there is **no rate limiting** of any kind on the accept loop.

**Fix**: `enter` should require auth before auto-create. Anonymous clients can `enter` an existing board but not create new ones. Alternative: queue rate-limiting as a 0.8 v1-hardening item.

#### M5 — Per-board policy default `open` admits anonymous CLI posts at v0.5.x stores

**Where**: `src/board.cyr:933-955` (`board_can_post`), upgrade-path on stores that pre-date 0.6.0.

A 0.5.x store has no `.policy` files. After 0.6.0 install, `board_policy_get` returns `POLICY_OPEN` for every board. `board_can_post` for `POLICY_OPEN` requires only a non-empty `session_fp` — so anonymous wire posts are denied (good). But **CLI anonymous posts** also pass the check incorrectly: `cmd_post` with no `--as` passes `from_fp = 0` to `board_can_post`, which returns 0 (denied at `:936`). So CLI anon is denied. ✅ Actually correct.

Re-checking the comment on `src/main.cyr:1264-1271`: yes, the CLI path correctly emits "board policy requires authentication" for anon CLI posts.

**Re-categorized**: NOT A FINDING. Filing as confirmed-correct after re-read; documenting the trace for the closeout pass.

#### M6 — Login challenge has no explicit deadline

**Where**: `src/main.cyr:548-591` (`login` command), MODE_LOGIN_AWAIT_SIG handler.

ADR 0006 § Specifics specifies a 30 s deadline on a parked challenge. M6-C deferred this — it relies on `RECV_TIMEOUT_SECS = 60` to drop quiet sockets. But that's the WHOLE-SOCKET idle timeout; a misbehaving client can hold a parked challenge open by sending heartbeat bytes (any data byte resets the recv timer).

Today: low impact because `g_login_fp` is overwritten by the next `login` invocation or cleared on session close. No state poisoning, no resource leak beyond the 96-byte slot.

Tomorrow (concurrent-accept): a client could indefinitely block a slot in a per-conn auth state machine.

**Fix**: capture a monotonic timestamp at `login` time; in MODE_LOGIN_AWAIT_SIG dispatch, if `now - login_started > 30s`, clear the challenge and emit "login timed out" before processing the `auth:` line. **This was already on the 0.7.x queue from M6-close — landing here.**

### LOW

#### L1 — `keyfile_load_seed` does not enforce file mode 0o600

**Where**: `src/account.cyr:327`.

`keyfile_generate` creates the keyfile with `PERM_KEYFILE = 0o600`. But `keyfile_load_seed` doesn't `stat` the file to verify mode is still 0o600 — a user who copies the keyfile to a world-readable location, or runs `chmod 644` accidentally, gets no warning. Other tools (`ssh`) refuse to load over-permissive keys.

**Fix**: defense-in-depth — `fstat` the keyfile after open, warn on stderr if mode & 0o077 != 0. Don't refuse (some operators legitimately want world-readable in containerized deployments).

#### L2 — Telnet `whoami` reveals bound identity to anyone on the same session

The `whoami` command emits `<handle> <fp16>` without auth check. By design — the session's identity is bound on `login`; revealing it back is the entire point. Not a finding; documenting that it's intentional.

**Status**: NOT A FINDING (working as designed per ADR 0006).

#### L3 — `account_resolve_handle` O(n) scan + bump-alloc per call

Each login attempt scans `<store>/.users/` and bump-allocates a `handle_buf` per iteration via `account_lookup_handle`. At 10s-of-users this is fine. At 1000+ users this allocates ~33 KB per login attempt (1000 × 33-byte `raw[34]` stack buffers — those are stack, OK — plus the `users_path` + `handle_buf` allocs which are one-each at ~544 B).

**Status**: well within v1.0 scale; queue for v2.x federation when user counts grow.

#### L4 — Loose command-prefix matching

`line_has_prefix(line, len, "login")` matches `loginfoo`, `logins`, `login123`, etc. None of these are exploitable for auth bypass (parsing always fails on the trailing bytes), but it's UX-confusing. Same for every other verb. **Status**: cosmetic / UX; not security.

#### L5 — `parse_auth_sig` retains `hex_decode` heap-allocated scratch

`hex_decode` returns a heap buffer; `parse_auth_sig` `memcpy`s into `sig_out` and drops the reference. If `hex_decode` is bump-allocated, that's 64 bytes leaked per login attempt. Symptom of the broader M1 bump-alloc lifecycle, not a separate finding.

**Status**: subsumed by M1.

### DOCUMENTED (known trade-offs)

#### D1 — Plaintext telnet wire

ADR 0006 § Negative documents: no TLS at M6; an active MITM can hijack a session post-auth. Authentication is sound under passive observation (challenge nonces are single-use + random) but cannot defend against active wire attackers. Pre-1.0 acceptable; v2.x pillar 1 (federated identity) will require a transport story.

#### D2 — 64-bit fingerprints (collision birthday-bound at 2³² users)

ADR 0006 § Consequences. Per-deployment scale (10s of users) makes collision risk negligible. Extension to 32 hex chars is non-breaking forward-compat.

#### D3 — Bump-allocator memory model

CLAUDE.md § Cyrius Conventions. `alloc()` is bump-allocated with no `free()`. Long-running `serve` accretes memory; the planned mitigation is per-connection arenas in the concurrent-accept refactor. **Surfaced as M1 above** because the magnitude (~73 KB / conn + ~100 KB / read) makes it a real DoS vector even at modest scale.

## 0.7.x fix slate

Landing in 0.7.x:

1. **H1** — CLI subject byte-validation (no CR / LF / NUL / ESC / other C0 control bytes except TAB).
2. **H2** — `cmd_list` + `cmd_read` `--board` validation via `board_name_valid`.
3. **H3** — `post_from` re-validates handle + fp shape on read; malformed → treat as anonymous.
4. **M3** — `parse_post_id` caps prefix at 18 digits.
5. **M6** — explicit 30 s deadline on parked login challenge (was on M6-close deferred queue).

Each fix lands with a regression test in `src/test.cyr` (t71+).

## Deferred to 0.8 v1-hardening

- **M1 + M2** — concurrent-accept refactor with per-connection memory arenas + per-conn auth state slots.
- **M4** — `enter` auth requirement OR accept-loop rate limiting.
- **L1** — keyfile mode warn-on-load.
- sigil 3.1.1 → 3.4.3 release-notes diff read.
