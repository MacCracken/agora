# 2026-06-15 — agora 1.4.4 P(-1) hardening / security audit

> **Audit window**: agora 1.4.3 (`ac3e53a`) — post-decode-arc, pre-1.4.4 hardening cut.
> **Scope**: full re-audit on the **6.2.7 toolchain** + **darshana 0.7.0** bump. Telnet wire core (`src/telnet.cyr`, the ingress paths in `src/main.cyr`), persistence/filesystem surface (`src/board.cyr`, `src/account.cyr`, `src/door.cyr` save/world I/O), every door-game input parser + shared RNG/wager/compete primitives, and a cross-cutting internal review (dead code, refactor, optimization, save/load clamp consistency).
> **Method**: four parallel line-by-line reads against the CLAUDE.md "Security Hardening" checklist + the project's Cyrius footgun list; review of CVE-2020-10188 (telnetd subneg overflow) and CVE-2011-4862 (encrypt key overflow) for parity; cross-check against the 2026-05-23 (0.7.0) audit for regressions.
> **Baseline**: clean DCE build (`CYRIUS_DCE=1 cyrius build`), 218/218 tests pass; worldbench 20k flock'd txns in 476 ms (~42k txn/s).

---

## Severity rubric

- **CRITICAL**: remote compromise / privilege escalation from a single bad input.
- **HIGH**: remote DoS or auth bypass with moderate effort, or memory corruption from untrusted input.
- **MEDIUM**: requires specific conditions (tampered store, operator misconfig, crafted economy input).
- **LOW**: defense-in-depth — current code is correct, a near-future change could regress.
- **DEFERRED**: real finding whose fix lives outside agora's editable surface (stdlib) or is a deliberate non-goal for this cut.

---

## Findings

### CRITICAL

*None.* No remote-compromise path found. The IAC parser bounds-checks every state transition, subneg accumulation is hard-capped (CVE-2020-10188 parity holds), auth uses fresh single-use CSPRNG nonces, and post/save storage uses `O_EXCL` + per-board/per-world flock with the alphabet validators as the sole path-traversal defense.

### HIGH

#### T1 — `send_buf` infinite busy-loop on partial-write error (remote CPU-peg DoS) — **FIXED**

**Where**: `src/main.cyr:229` (`send_buf`).

The send loop was `while (off < len) { if (done != 0) { continue; } var n = sys_write(...); if (n <= 0) { done = 1; continue; } off = off + n; }`. When `sys_write` returned `<= 0` (peer reset mid-send) after a prior partial write left `off < len`, it set `done = 1` and `continue`d — but `off < len` was still true, so the loop re-entered, hit `if (done != 0) continue;`, and spun forever pegging a CPU core. This is the CLAUDE.md "no `break` in a `var`-declaring while loop — use flag+continue" rule applied *incorrectly*: the flag suppressed work but never terminated the loop. Reachable from every ingress path (IAC-response drain, `echo_byte`, MOTD/prompt sends).

**Trigger**: connect, send a byte to start the server echoing/replying, then RST the socket so a subsequent `sys_write` fails mid-buffer. The forked child spins at 100% until killed; fork-per-connection isolates but does not stop the aggregate burn across many connections.

**Fix**: drop the `done` flag; `return 1` directly on a short/failed write (`return`, unlike `break`, is a safe loop exit in Cyrius). Loop terminates correctly; caller still closes the connection on a non-zero return.

#### S1 / R4 — `th_load_parse` unbounded registry count → heap overflow from a tampered save — **FIXED**

**Where**: `src/handler.cyr:689` (`th_load_parse`).

The cable count was read straight from the save and used as the write-loop bound with no clamp: `var n = save_get_i64(...); store64(st + TH_REGCOUNT, n); while (j < n) { var cb = th_cable_at(st, j); ... }`. `th_cable_at` does raw pointer arithmetic (`TH_REG_P + j*CB_SIZE`) into an `alloc(TH_REG_MAX * CB_SIZE)` = 256×80 = 20,480-byte buffer with no range check. A handler save with `n` > 256 overruns the heap by `(n − 256)×80` bytes of attacker-influenced i64 data. The `DOOR_SAVE_CAP` read limit (32,768 B) still permits ~3,000 cable records — far past index 256 — so the overflow is real and large.

This is the **lone divergence** from the project's load-clamp discipline: `ol_load_parse` and `decode_load_parse` already clamp their variable counts; `jw_load_parse` clamps the learned-pair count; the handler simply missed the guard.

**Trigger**: edit `<store>/.users/<fp16>/games/handler.sav` to set the cable count line to e.g. `4000`, then enter handler (solo or Universe) as that user.

**Fix**: clamp before the write loop — `if (n < 0) { n = 0; } if (n > TH_REG_MAX) { n = TH_REG_MAX; }` — mirroring `ol_load_parse`/`decode_load_parse`. Severity HIGH not CRITICAL because exploitation needs write access to the save file, not a pure wire input. Regression-pinned by **t219** (hostile count → clamped; negative count → 0).

### MEDIUM

#### D2 — unbounded integer accumulation in `door_parse_int` / `door_parse_2int` → economy overflow — **FIXED**

**Where**: `src/door.cyr:334` (`door_parse_int`), `:361` (`door_parse_2int`).

Both accumulate `val = val*10 + digit` with no magnitude cap. Every client-entered quantity/stake/amount flows through them (smuggler buy/sell qty + bank/shark amounts, quest deposit/withdraw + card stake, handler fund, olympiad stake). The exposed case is `sl_buy`'s `var cost = price * qty`: a crafted `qty` near i64-max makes `cost` wrap negative, passing the `cost > cash` guard and letting a player "buy" goods while their cash *increases*. The wager path itself is safe (`wager_resolve` has an explicit overflow guard), but the upstream parser was the shared root cause.

**Fix**: pin the accumulator at `DOOR_INT_MAX` (1e9) inside both parsers — far above any real economy figure, and `1e9 * 1e9 = 1e18 < i64 max`, so no single downstream multiply can wrap. Regression-pinned by **t220**.

#### D1 — integer overflow in `ash_scan_ints` order quantity — **FIXED**

**Where**: `src/ashes.cyr:1090` (`ash_scan_ints`).

Same unbounded `v = v*10 + digit` pattern in the war-game order parser. Combat itself clamps delivery (`deliver = imin(qty, budget)`), so the damage was bounded, but the unclamped qty was stored in the shared snapshot and rendered to every viewer. **Fix**: pin at `DOOR_INT_MAX` (the same constant), closing the D1+D2 class at its two roots.

### LOW

#### R6 — `door_world_commit` dropped the `world_write` return — **FIXED**

**Where**: `src/main.cyr:1262`. A failed shared-world snapshot write (e.g. ENOSPC under the held lock) was silently lost. **Fix**: capture and return the `world_write` status (lock still released on either path, so an error can't strand the world locked).

#### T3 — `TERMINAL_TYPE` subneg buffer has no NUL terminator and clamps with `>` not `>=` — **not changed (defense-in-depth)**

**Where**: `src/telnet.cyr:642`. The boundary is *correct today* (`payload_len == TERM_TYPE_CAP` copies exactly in-bounds, no overflow — this is the agora analogue of CVE-2020-10188 and it is defended). Flagged only because a future consumer treating `tt` as a cstring would read past the stored length. Left as-is for 1.4.4; recorded for the v1.0 freeze.

#### D3 — door render functions accept a `cap` but never check it — **not changed (defense-in-depth)**

**Where**: every `*_render` in the door modules. Current worst-case frames (Handler roster, Ashes 12-province map + ≤64 orders, Olympiad lanes) all fit comfortably under the fixed `DOOR_FRAME_CAP` (8192), so there is no overflow today — but the "render output < 8192" invariant is unenforced. Recorded; revisit if any frame's row count grows.

### DEFERRED

#### T2 — no send-side socket timeout (slowloris-read) — **DEFERRED to lib/net.cyr**

**Where**: `handle_client` sets `SO_RCVTIMEO` but there is no `SO_SNDTIMEO` setter. A client that never drains its receive window parks a forked worker in a blocking `sys_write`; the recv timeout doesn't fire because the worker is in `write`, not `read`. Fork-per-connection bounds the blast radius to one process per connection (up to `RLIMIT_NPROC`).

`lib/net.cyr` exposes only `sock_set_recv_timeout` — there is no send-timeout primitive, and `lib/net.cyr` is a toolchain-synced module (gitignored, regenerated by `cyrius lib sync`). Per CLAUDE.md ("the listener uses `lib/net.cyr` socket primitives, not raw syscalls; new platform support is a `lib/net.cyr` task"), adding `SO_SNDTIMEO` is a **net stdlib task**, not an agora edit. Tracked here for the net.cyr backlog; agora should set the send timeout alongside the recv timeout once the primitive lands.

## Internal review / optimization (non-security)

- **Static-data warning (325 KB)** — investigated and **not actionable as agora source**. The largest `var buf[N]` in the entire codebase is 144 bytes; every buffer ≥4 KB already routes through `alloc()`/`fl_alloc()`. The 325 KB is the linked static section (string literals + the broad 6.2 stdlib surface the project deliberately kept after the lib reorg). The lever, if ever pulled, is trimming unused `stdlib` modules from `cyrius.cyml` — a deps decision, not a code change. No action for 1.4.4.
- **Dead code (R3)** — removed 7 genuinely-orphaned functions (0 non-definition refs in src or tests): `ez_is_over`, `py_is_over` (never-wired door-contract symmetry), `qu_clampv` (dead + a per-file dup of `iclamp`), `rng_seed_from_clock`, `th_format_standing`, `th_has_mole`, `paw_stock_cells`. Binary −720 B; unreachable-fn count 1492 → 1485.
- **Door-dispatch consolidation (R7) — DEFERRED.** `main.cyr` carries ~47 `if (g_door_game == GAME_X)` arms across ~18 `door_*` functions; well past the "third instance" refactor threshold, but a vtable/descriptor-table collapse is a Large task and the dispatch is stable + purely additive. Deferred per the no-speculative-refactor policy — revisit when the 11th door lands.
- **Save/load clamp consistency** — verified all 8 `*_load_parse`: 7 already clamp or have only fixed-constant loops; the handler (S1/R4) was the sole outlier and is now fixed, so the discipline is uniform.

## Verified-clean checklist items

- IAC option-code + subneg payload bounds-checking (every `load8` length-guarded; option codes masked `& 0xFF`).
- CVE-2020-10188 parity: `ts_sb_push` hard-caps subneg accumulation at `SB_BUF_CAP` (512), drops overflow.
- CVE-2011-4862 parity: N/A — no ENCRYPT option; unknown options naive-refused, unknown subnegs dropped.
- Path traversal: `board_name_valid` + `game_name_valid` + `fp16_valid` gate every external path component; H2 (0.6.0 `--board` traversal) fix verified present, not regressed.
- Header/CRLF injection: CLI subject validated by `header_text_cstr_ok`; wire subject can't carry CR/LF (line-framed at ingest). H1 fix verified.
- No `sys_system` anywhere; post bodies/paths inert.
- `O_EXCL` + flock on post claim-and-write, keyfile gen (0600), world txn read-modify-write. No TOCTOU found.
- RNG fairness: `wager_uniform_csprng` / `ol_compete_*` read the kernel CSPRNG, refuse short reads, never seed/store/replay the result draw; payout odds computed server-side from forms, not client input.
- Economy overflow: `wager_resolve`/`wager_payout`/`wager_settle` carry explicit overflow guards (the strongest code in the set); the gap was upstream in the shared parser (D1/D2), now closed.
- No `break` in `var`-declaring while loops (the flag+continue idiom is respected throughout; T1 was a *misuse* of it, not a `break`).
