# 0014 — War-game door — asynchronous shared-world strategy as the MUD on-ramp

> **Status**: Accepted — **in progress 1.3.7** (the war-game door, "Ashes of Empire"), the last door before **1.4.0 Descent link**.
> **Date**: 2026-06-08 (accepted 2026-06-09, bite 1: the shared-world snapshot foundation)

## Context

The 1.3.4–1.3.7 arc builds the casino mechanic and two new doors in a deliberate sequence (roadmap § *1.3.4–1.3.7*): the shared **wager** module (1.3.4), its integrations across the existing doors (1.3.5), a **training sim** that reskins QUEST's daily-turn economy (1.3.6), and finally a **war-game** (1.3.7). The first three are low-risk — they reuse abstractions agora already has (the `door.cyr` PRNG, the world-transaction framework, `qu_day_tick` rationing). The war-game is the one that earns its slot for a different reason.

The next minor after this arc is **1.4.0 — the Descent link** (roadmap § *1.4.0*): a logged-in agora session bridges into the sibling **Yeoman's Descent** MUD over the shared telnet substrate. The MUD is a *real-time room/object world* — the one thing agora's architecture has never had to support. Everything agora ships today is either asynchronous (boards, posts, the Universe doors resolve at *your* turn from a snapshot — [ADR 0010](0010-persistent-universe.md) § Decision: "Indirect / asynchronous PvP, not real-time") or single-process-live (chat tails a `flock`'d transcript on a poll tick — [ADR 0011](0011-chat-area.md)). Real-time *shared mutation between concurrent actors* is the MUD's hard problem, and agora has never rehearsed it.

The hard constraint that shapes every shared-state feature shapes this one too:

- **Fork-per-accept ([ADR 0007](0007-fork-per-accept-concurrency.md)) means no shared memory across sessions.** Each connection is its own process; the `g_*` session globals are private per child. Two players touch the same state **only on disk**, serialized with `flock` (ADR 0007 § Negative: "No shared state across sessions"). agora has **no daemon today** — it is one fork-per-accept binary, and [ADR 0010](0010-persistent-universe.md) § Alternatives *deferred* a long-lived world coordinator (a major moving part with its own lifecycle and supervision burden) rather than foreclosing it. This ADR keeps that door open: a resolution daemon is a deliberate **future** option (see Alternatives), not the first cut.
- **The world-transaction framework already exists.** 1.2.0 ([ADR 0010](0010-persistent-universe.md)) shipped `door.cyr`'s `world_lock_acquire` / `world_read` / `world_write` / `world_txn_add` — the `lock → read snapshot → pure transform → write → unlock` discipline, proven by the multi-process concurrency smoke (`08-world-concurrency.sh`). It currently carries per-player Universe state keyed by sigil fp (`hex16_to_i64`), depletable port stock, planet ownership, and async-PvP garrisons.
- **The lazy daily-tick-on-entry pattern already exists.** QUEST's `qu_day_tick` (`src/quest.cyr`) resolves a wall-clock day rollover *on the next caller's entry* — refilling rations and paying bank interest — using the real-time epoch (`clock_epoch_ns() / QU_NS_PER_DAY`), explicitly *not* a background process. A backward clock or a server restart self-heals rather than freezing (`qu_start` comment: epoch days survive a reboot, a monotonic index would freeze the refill).

The open question 1.3.7 answers: **what is the war-game door actually for, and how does it resolve batched multi-player turns on a shared map under fork-per-accept with no daemon — such that building it retires the concurrency risk the MUD is about to take on for real?**

## Decision

**The war-game's job is a new-capability proof, not genre-filling: it is the deliberate rehearsal of shared-state mutation between concurrent callers, extending ADR 0010's world transaction to a shared map + alliance state, with turn-batch combat resolved lazily on the next caller's entry. It is sequenced last of 1.3.4–1.3.7, immediately before 1.4.0, on purpose — it is the on-ramp to the MUD.** Like every shared-state surface, it requires login.

- **The door's purpose is the proof.** Asynchronous shared-world strategy — multiple players acting on a **common map** between sessions, forming **alliances**, with **turn-batched combat resolution** — is the conceptual bridge between the single-user doors (QUEST, the training sim) and the concurrent Descent MUD. It deliberately exercises the **shared-state-mutation-between-callers** path the MUD will later do in *real time*. So this door **retires a concurrency risk**; it is not on the roadmap to fill out the genre. Building it is how we learn the lessons that feed 1.4.0 *before* the MUD has to learn them live.

- **Extend ADR 0010's framework; do not reinvent it.** The war-game runs on the exact `door.cyr` world transaction the Universe doors use — `world_lock_acquire` → `world_read` → pure transform → `world_write` → `world_lock_release`. What scales is the *contents* of the snapshot, not the discipline: from per-player Universe state to a **shared map** (territory, terrain, contested fronts) + **per-player territory/army** (keyed by `hex16_to_i64` sigil fp, exactly as PA keys planet ownership) + **alliance / diplomacy state** (who is allied with whom, pending offers). Same lock → read → pure-transform → write. The transform stays a pure, unit-testable `(world, player, action) → (world', result)` — the most valuable property we have, preserved across every door since ADR 0009.

- **Turn-batch resolution is lazy, on the next caller's entry — no daemon.** Under fork-per-accept with no background process, the question "who runs the batch resolution?" has one answer that fits agora: **the next caller does, on entry, inside the world transaction**. Mirroring QUEST's `qu_day_tick` (resolve-overdue-on-entry, driven by `clock_epoch_ns`), entering the war-game checks whether the current turn window has elapsed; if so, the entering session — already holding the world lock — runs the pure batch-resolution transform over all queued orders, advances the turn counter, and writes the resolved world back before showing the player their position. A backward clock or a restart self-heals the same way QUEST's tick does. Orders queued during a window are batch-resolved together at the boundary, so combat is **adjudicated against the orders everyone left behind** — never a live socket-to-socket exchange, exactly the async-PvP shape ADR 0010 established. This is the *first cut*, not a foreclosure: a timer-driven resolution **daemon** that advances windows on a schedule regardless of traffic is kept explicitly open as a deliberate **later feature** (see Alternatives) — the lazy on-entry pattern is simply the right first step, and the right one to learn from before the MUD.

- **Async-PvP notification reuses the QUEST/PA prototype.** "Here is what happened to your territory while you were gone" is a next-login notice — the same next-login-notice shape QUEST/PA already prototyped (async resolution against assets the player left behind). No new mechanism; the resolved-world delta surfaces as a message on entry, after the lazy batch tick runs.

### Why last, and why before the MUD

The sequencing is the decision. The wager module (1.3.4) and its integrations (1.3.5) build a shared abstraction once; the training sim (1.3.6) reskins a shipped economy. None of those touches concurrency. The war-game is placed **last of the four and immediately before 1.4.0** precisely so the shared-mutation lessons — snapshot sizing under the flock, coarse-vs-fine locking under contention, batch-resolution cadence, conflict resolution when two players' orders collide — are learned on a door we fully control, on the proven `flock`'d framework, **before** the MUD takes those same problems on in real time across a second codebase's world loop. This is the last rehearsal; 1.4.0 is the performance.

## Consequences

- **Positive** — retires a real concurrency risk on the way to the MUD: the shared-state-mutation-between-callers path gets exercised on agora's own proven `flock`'d framework before 1.4.0 depends on it. Zero new dependencies and zero new concurrency *model* — it is ADR 0010's world transaction with a richer snapshot. The pure-transform rule survives, so map evolution, combat, and diplomacy stay unit-testable without a socket (only the interleaving needs a multi-process smoke, as in ADR 0010). The lazy resolve-on-entry pattern is already proven by QUEST and needs no daemon, no supervision, no new process lifecycle. World state stays operator-visible flat files (debuggable, backup-able). Genuinely good multiplayer strategy as a side effect.

- **Negative** — the richest snapshot agora has yet stored under one coarse `flock`: a full shared map + per-player armies + alliance graph in a single locked blob, which contends harder under fanout than the per-commodity stock PA stores, and makes **world-format versioning** (already load-bearing since ADR 0010 § Negative) carry more weight. Lazy resolution means a turn window only advances when *someone enters* — an idle world's batch can sit unresolved past its wall-clock boundary until the next caller triggers it (acceptable: the resolution is deterministic from queued orders + the elapsed-window count, so the delay is cosmetic, never a correctness issue — same self-healing property as `qu_day_tick`). The batch-resolution transform is the most complex pure function in the door tree and the conflict-resolution rules (colliding orders, simultaneous claims) are new surface to get right.

- **Neutral** — the war-game requires login, like Universe and chat (a persistent territory is a persistent actor); anonymous players keep Practice/Solo doors and read-only boards. The shared map + per-player state + alliance state are separate concerns within one snapshot, with one lifecycle (the world tree under `<store>/.games/<game>/world/`), pruned by the operator like any other world. Whether the lazy-tick lesson transfers cleanly to the MUD's real-time loop is exactly what 1.3.7 exists to find out — that uncertainty is the point, not a gap.

## Alternatives considered

- **A long-lived resolution daemon that ticks the world on a timer** — advances turn windows on schedule regardless of traffic, and is the natural stepping-stone toward the MUD's real-time loop. **Deferred, not rejected — kept explicitly open as a later feature.** It is not the *first* cut for two reasons: it would be agora's first long-lived process, carrying the lifecycle / supervision / crash-recovery burden [ADR 0010](0010-persistent-universe.md) § Alternatives deferred for the world coordinator, and it steps outside "agora is one fork-per-accept binary" ([ADR 0007](0007-fork-per-accept-concurrency.md)). Shipping the lazy resolve-on-entry pattern (QUEST's `qu_day_tick`) first retires the shared-state risk with **no** new process; a daemon can then layer on once the model is proven — and when it does, it earns its own ADR + a supervision story. Indeed, when agora eventually grows toward the MUD's real-time cadence, a resolution daemon is the **likely first long-lived process** — so this is a road kept deliberately open, just not walked first.

- **Real-time turn resolution (orders resolve the instant they are issued, live)** — closer to a strategy game with simultaneous players, but that *is the MUD's job*. Building real-time shared mutation into a BBS door would either re-implement the coordinator we just rejected or fight the line-oriented, fork-per-accept session loop the whole tree is built on. Keeping the boundary — async here, real-time in Descent — is deliberate: the war-game rehearses the *shared-state* problem without prematurely importing the *real-time* problem. Rejected.

- **Skip the war-game; go straight from the training sim (1.3.6) to the MUD link (1.4.0)** — saves a slot. Rejected: the MUD would then be agora's *first* contact with shared-state mutation between concurrent actors, on an unfamiliar second codebase's world loop, with no rehearsal on the framework we control. The war-game's whole value is being that rehearsal — it is a risk-retirement bite, and dropping it moves the risk into the harder release.

- **A fresh per-game concurrency layer tuned for the shared map** — bespoke locking sized for map/army/alliance access patterns. Rejected for the same reason [ADR 0010](0010-persistent-universe.md) § Alternatives rejected per-game bespoke concurrency: divergent locking is multiplied surface for race bugs. One world-transaction framework, all the doors on top — and reusing it *is* the rehearsal value (we harden the shared framework, not a throwaway).

- **Fine-grained locking from the start (per-territory / per-front locks)** — would cut contention on the big snapshot up front. Rejected as premature, consistent with [ADR 0010](0010-persistent-universe.md) § Decision ("Coarse lock first … Per-sector / per-record locks are a later optimization if contention shows up; they bring lock-ordering hazards, so they wait for evidence"). Ship the coarse single-world `flock` first; measure contention under real fanout; refine only on evidence. Whether the shared map forces this decision earlier than PA did is an open question below.

## Open questions

- **Shared-map snapshot size + coarse-vs-fine locking under contention.** A full map + per-player armies + alliance graph is the largest blob agora will hold under one `flock`. Does the coarse single-world lock ([ADR 0010](0010-persistent-universe.md)) hold up at fanout, or does the map force per-territory / per-front locking (and its lock-ordering hazards) sooner than PA's per-commodity stock did? Measure before refining.
- **Resolution cadence + who triggers it.** Wall-clock tick (a fixed turn window, resolved lazily on the next entry past the boundary — the leading candidate, mirroring `qu_day_tick`) vs. on-Nth-action (resolve once N orders are queued) vs. operator-run (an admin verb advances the turn). The lazy on-entry trigger is favored *for the first cut* because it needs no daemon, but an idle world advancing only on the next visit is a real behavioral choice to nail down — and a **timer-driven daemon** is the open future option that would decouple cadence from traffic entirely (deferred, not foreclosed — see Alternatives), the same path agora would walk toward the MUD's real-time loop.
- **The alliance / diplomacy model.** How alliances form, bind, and break in the snapshot; whether allied orders resolve cooperatively in the batch; how betrayal and pending offers are represented and adjudicated when colliding orders meet. This is the genuinely new game-design surface, and it is where the conflict-resolution rules of the batch transform get hard.

This is the last rehearsal before the MUD's harder real-time concurrency in 1.4.0. Every open question above is one we would rather answer on a door we fully control, on the proven `flock`'d framework, than discover live in the Descent link.
