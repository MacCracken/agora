# 0011 — Chat area (live multi-user teleconference)

> **Status**: Accepted — implemented in 1.3.0 (in progress). Bite 1: the chat surface (channels, `flock`'d ring transcript, `MODE_CHAT`, live-tail poll). Eliza (the anchor inhabitant) lands as a separate concern in the same minor — a pure module reachable as a `play eliza` door and a private `/eliza` side-channel inside chat ([ADR 0009](0009-door-games-subsystem.md) shape, no new deps).
> **Date**: 2026-06-08

## Context

agora has shipped the *asynchronous* public-assembly surface — boards, threaded posts, durable artifacts (ADRs 0002–0005). The *synchronous* surface — the classic BBS **teleconference** / CB-simulator where logged-in citizens talk in shared named channels in (near) real time — is the one civic-marketplace primitive still missing. 1.3.0 builds it.

The same hard constraint that shaped Universe ([ADR 0010](0010-persistent-universe.md)) shapes chat:

- **Fork-per-accept ([ADR 0007](0007-fork-per-accept-concurrency.md)) means no shared memory across sessions.** Each connection is its own process; the `g_*` session globals are private per child. Two chatters touch the same state **only on disk**, serialized with `flock`. So chat is *not* a new concurrency model — it is the 1.2.0 world-transaction discipline pointed at a transcript.
- agora's session loop is **line-oriented and blocking-read** (`handle_client`): a session only wakes when the client sends bytes. A live chat needs lines to arrive *without* the reader typing. The one lever already present is `sock_set_recv_timeout` (the slowloris defense): a recv that times out returns `-EAGAIN` (`-11`), distinct from EOF (`0`) and real errors. That timeout is the tick clock chat tails on.

The open question 1.3.0 answers: **how does a shared, live-updating chat room live on disk under fork-per-accept and a blocking-read line loop, without a daemon, without losing the on-disk-is-the-only-shared-state discipline?**

## Decision

**A per-channel append-only `flock`'d ring transcript that sessions poll-tail on a short recv-timeout tick; live-tail by sequence number, not byte offset.** Chat requires login.

- **Storage.** `<store>/.chat/<channel>/log` is the transcript; `<store>/.chat/<channel>/.lock` is the `flock` target. The `.chat/` tree is dot-prefixed so it dodges board/post enumeration exactly like `.users/` and `.games/`. Channel names validate through the existing **game-name alphabet** (`game_name_valid`: 1–32 bytes, `[a-z0-9]` lead, `[a-z0-9-_]` tail, no `/`, no leading dot) — the single path-traversal guard, same as the save/world paths. (Boards reserve `main`; channels do not — separate namespace.)
- **Transcript line = `<seq>\t<handle>\t<text>\n`.** `seq` is a per-channel monotonic counter; `handle` is the sigil-bound identity; `text` is sanitised to printable ASCII (32–126), which drops CR/LF/TAB/ESC/NUL/high bytes — so a message can neither break the line format (no embedded tab/newline) nor inject terminal escapes into another viewer's session (the same ESC threat board.cyr's ingress filter defends). Each field is a pure, unit-testable transform (format / parse-seq / render).
- **Say = a world transaction (lock → read → trim → append → write → unlock).** `chat_say` takes the channel lock, reads the current ring, derives the next `seq` from the last line, formats the new line, appends, **trims whole lines off the front until the ring is ≤ `CHAT_LOG_CAP` (32 KB)**, and writes the whole ring back under the held lock (the bite-1 `world_write` `O_TRUNC` pattern — correct against cooperating lock-holders). Two sayers serialise on the `flock`; sequence numbers stay unique and monotonic.
- **Live-tail by sequence, not byte offset.** Each session remembers the highest `seq` it has shown (`g_chat_seen`). On entry it prints the last `CHAT_SCROLLBACK_LINES` (16) as scrollback and sets `seen` to the latest seq. Thereafter, on every tick it re-reads the (bounded ≤ 32 KB) ring and renders only lines with `seq > seen`, advancing `seen`. **Sequence numbers are absolute**, so the front-trim that bounds the ring never invalidates a reader's position the way a byte offset would — the central reason the ring uses a seq counter instead of an `lseek` offset (the stdlib has no Linux `lseek` wrapper anyway). A session idle long enough that unseen lines were already evicted simply misses them — chat is live presence, not a guaranteed-delivery queue.
- **The tick is the recv timeout.** Entering chat lowers the socket recv timeout to `CHAT_POLL_SECS` (2 s). In `handle_client`, an `n == -EAGAIN` read **while `mode == MODE_CHAT`** is a poll tick — flush new lines and continue — rather than the disconnect every other mode treats it as. To avoid garbling a half-typed line, a tick only flushes when the input buffer is empty (`line_pos == 0`); mid-typing, new lines wait for the next idle tick or the user's Enter. A separate accumulated-idle counter bounds true silence at `CHAT_IDLE_SECS` (15 min) so an idle chatter can't pin a forked worker forever; leaving chat restores the 60 s command-mode timeout.
- **Commands inside chat begin with `/`** (`/leave`, `/help`, `/quit`); any other line is said to the room. The leading-`/` convention reserves the namespace for `/eliza` (Bite 2) and future `/who`, `/me` without colliding with message text.

### Eliza (same minor, separate concern)

Eliza is the room's anchor inhabitant but **not** part of the chat transcript machinery. She is a pure module (`src/eliza.cyr`, ADR 0009 shape: `(input line) → (reflected response)` — keyword-ranked decomposition + reassembly templates + pronoun reflection), reachable two ways: a `play eliza` **door** (solo, no socket knowledge), and a private **`/eliza` side-channel** inside chat that answers only the addressing user and writes **nothing** to the shared transcript (chosen over an always-present room-bot: no write-arbitration story, no transcript noise, matches the "go talk to the shrink" BBS feel). The engine is factored as a reusable decomposition/reassembly core with a swappable script so **PARRY** (Colby, 1972) can drop in at 1.3.1 as `play parry` + `/parry` with no engine rework.

## Consequences

- **Positive** — the missing synchronous public-assembly surface, with zero new dependencies and zero new concurrency model: chat is the proven 1.2.0 `flock`'d transaction pointed at a transcript. Sequence-keyed tailing is rotation-safe by construction. The pure line transforms (format / parse / render / sanitise / ring-trim / tail-since) are all unit-testable without a socket; only the interleaving needs a smoke. Transcripts are operator-visible flat files (debuggable, prunable, backup-able).
- **Negative** — the live tail is a **poll**, not a push: a message appears up to `CHAT_POLL_SECS` after it is said, and a burst exceeding the per-tick render buffer (8 KB ≈ 30+ lines in 2 s) drops the overflow for that tick. A forked worker per idle chatter is the cost of fork-per-accept presence (bounded by `CHAT_IDLE_SECS`). Mid-typing, inbound lines are deferred to keep the input line clean — a deliberate cosmetic trade. The 32 KB ring is lossy scrollback, not durable history (by design — chat is ephemeral; boards are the durable surface).
- **Neutral** — chat requires login (a presence is a persistent-ish actor), like Universe; anonymous users keep read-only boards. `/who` presence (a `flock`'d heartbeat roster) and durable/rotated scrollback are deferred follow-ons, not gaps in the first cut. Channel lifecycle is implicit: a channel exists once its dir is created on first join and persists until an operator prunes it.

## Alternatives considered

- **Poll-on-input only (no timeout tick)** — flush new lines only when the reader presses Enter. Zero changes to the recv/disconnect path, but it is a *refresh-on-keystroke board*, not a live room: a lurker sees nothing until they type. Rejected — the recv-timeout tick is cheap and already-present, and liveness is the whole point of a teleconference.
- **Byte-offset tail (`lseek` to last position, read to EOF)** — the obvious tail, but (a) there is no Linux `lseek` wrapper in the stdlib, and (b) the front-trim that bounds the ring shifts every byte, so a stored offset silently desyncs after the first rotation. Sequence numbers are absolute and survive rotation. Rejected.
- **Unbounded append-only transcript + external rotation** — simplest writer, but whole-file reads (the only offset-free read available) grow without bound, and "durable artifacts" is the boards' job, not chat's. The 32 KB ring keeps every tick cheap and bounds memory. Rejected for the live surface; durable/rotated scrollback can be a later opt-in.
- **A long-lived chat daemon holding rooms in memory, sessions talking to it over IPC** — real push, no per-tick disk read, but a major moving part with its own lifecycle/supervision/crash story that breaks "agora is one fork-per-accept binary" — the same reason ADR 0010 deferred a world coordinator. The disk-transaction model is the right first cut; revisit only on a measured latency/contention wall.
- **Always-present Eliza room-bot writing into the transcript** — more social, but needs a write-arbitration story under fork-per-accept (who runs her turn?), pollutes scrollback, and risks feedback loops. The private `/eliza` side-channel ships the nostalgia with none of that; a callable room-bot is an explicit later add. Rejected for the first cut.
- **Real-time per-byte chat (character-at-a-time, no line buffering)** — closer to true CB, but fights agora's line-oriented LINEMODE/echo machinery (ADR 0009-era) and every existing mode. Line-at-a-time is the house idiom. Rejected.
